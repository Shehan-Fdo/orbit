# Orbit Developer Handbook (Start to Finish)

This document is a full, practical guide for setting up Orbit, understanding how it works internally, and taking changes from local development all the way to a production-ready build.

If you are new to this repository, follow sections in order. If you are already contributing, jump to the workflow and troubleshooting sections.

---

## 1) Orbit at a glance

Orbit is a Markdown-first static site generator written in Node.js (ES modules). It transforms a content tree under `src/pages` into a fully linked static site in `dist`.

### What Orbit generates

- Content pages from `.md` / `.mdx` files.
- Sidebar navigation derived from folders/files.
- Previous/Next pagination between content pages.
- Root index and per-course index pages.
- SEO artifacts:
  - `dist/sitemap.xml`
  - `dist/robots.txt`
  - `dist/feed.xml`
  - `dist/seo-report.json`

### Core runtime files and their responsibilities

- `main.js`
  - orchestrates full build
  - handles watch mode
  - generates SEO outputs and audits warnings
- `layout.js`
  - renders final page shell (head tags, sidebar slot, content slot, pagination)
- `utils.js`
  - parses naming prefixes
  - builds cleaned paths
  - recursively discovers markdown files
- `.orbit/.sidebar/sidebar.js`
  - builds content tree
  - resolves ordering rules
  - renders sidebar HTML
  - flattens pages for prev/next navigation
- `serve.js`
  - previews `dist` locally
  - supports clean URL directory routing

## 4) Project structure and data flow

### Source of truth

- `src/pages/**` -> content source
- `src/styles/**` -> copied to output styles folder
- `public/**` -> copied to output assets folder
- `metadata.json` -> global SEO defaults + runtime flags

### Build output shape

- `dist/index.html` -> root landing page
- `dist/content/<route>/index.html` -> generated content pages
- `dist/content/<course>/index.html` -> generated course index pages
- `dist/sitemap.xml`, `dist/robots.txt`, `dist/feed.xml`, `dist/seo-report.json`

### Important repository behavior

- `dist/` is ignored by git.
- `src/pages/` is currently ignored by git in this repository.
  - If collaboration requires content versioning, decide whether to adjust `.gitignore`.

## 5) How routing works (important for contributors)

Orbit creates folder-style routes by writing each content page to:

`dist/content/<clean-path>/index.html`

Clean path is generated by:

1. removing sorting prefixes from each path segment
2. removing `.md` / `.mdx`
3. appending trailing slash for internal route consistency

Examples:

- `src/pages/[1]__web-dev/[1.1]__selectors.md`
  -> route: `/content/web-dev/selectors/`
- `src/pages/01-intro.md`
  -> route: `/content/intro/`

The preview server redirects any `/index.html` request to canonical clean paths.

## 6) Ordering and sidebar behavior

Orbit supports multiple ordering strategies so editors can control navigation without hardcoding menu structures.

### File/page ordering priority (highest to lowest)

1. frontmatter `order`
2. parent folder `_meta.json` -> `items`
3. filename prefix (`[1.2]__name.md` or `01-name.md`)
4. alphabetical by title

### Folder ordering priority

1. folder `_meta.json` -> `order`
2. folder name prefix (`[2]__backend` or `02-backend`)
3. alphabetical by display label

### `_meta.json` capabilities

Place `_meta.json` in any folder under `src/pages` to:

- set folder display label (`label`)
- set folder sort order (`order`)
- override child file/folder order (`items`)

Example:

```json
{
  "order": 2,
  "label": "Backend Fundamentals",
  "items": {
    "api-basics": 1,
    "database-intro": 2
  }
}
```

## 7) SEO system: how values are resolved

Orbit resolves SEO fields per page using fallback layers:

1. frontmatter value
2. generated fallback (for example description excerpt from markdown body)
3. global defaults from `metadata.json`

### Global metadata fields (`metadata.json`)

- `siteName`
- `url` (critical for canonical URLs, sitemap, feed)
- `defaultTitle`
- `defaultDescription`
- `defaultImage`
- `author`
- `twitterCard`
- `twitterSite`
- `language`
- `keywords`
- `strictSeo` (fail build when warnings exist)
- `minifyHtml` (default true)

### Per-page frontmatter fields (recommended)

- content and navigation:
  - `title`
  - `order`
- SEO and social:
  - `description`
  - `image`
  - `keywords`
  - `canonical`
  - `twitterCard`
  - `twitterSite`
  - `schemaType`
  - `author`
  - `date`
  - `lastModified`

### SEO outputs generated each build

- canonical tags
- Open Graph tags
- Twitter tags
- JSON-LD structured data (`TechArticle` default)
- sitemap (`dist/sitemap.xml`)
- robots (`dist/robots.txt`)
- RSS feed (`dist/feed.xml`)
- warning report (`dist/seo-report.json`)

## 8) Day-to-day development workflow

## Step A: start watch mode

```bash
npm run dev
```

Watch mode rebuilds when these paths change:

- `src/pages`
- `src/styles`
- `public`
- `layout.js`
- `utils.js`
- `metadata.json`
- `.orbit/.sidebar/sidebar.js`
- `.orbit/.sidebar/sidebar.css`

## Step B: preview built output

In another terminal:

```bash
npm run preview
```

Default URL:

- `http://127.0.0.1:5500`

Custom port:

```bash
PORT=8080 npm run preview
```

PowerShell:

```powershell
$env:PORT=8080; npm run preview
```

## Step C: validate before commit

```bash
npm run lint
npm run seo:check
npm run build
```

Then inspect:

- generated pages in browser
- `dist/seo-report.json`

## 9) Content authoring conventions

These conventions prevent navigation and SEO regressions.

### File naming

Prefer one of:

- `[1.2]__topic-name.md` (recommended for nested order clarity)
- `01-topic-name.md` (legacy supported)
- `topic-name.md` (alphabetical fallback)

### Folder naming

Prefer consistent slug style (`kebab-case`) so route output remains predictable.

### Frontmatter baseline template

```yaml
---
title: Topic Title
description: Short, specific summary of what the page teaches.
order: 1
image: /assets/topic-cover.png
keywords:
  - keyword one
  - keyword two
date: 2026-04-08
lastModified: 2026-04-08
---
```

### Content quality checklist

- One clear H1 idea per page
- Description length roughly 70-180 chars
- At least one social image for important pages
- Valid canonical for externally visible pages

## 10) Build internals (for maintainers)

Build pipeline in `main.js` runs in this order:

1. load global metadata
2. clear and recreate `dist/content`
3. copy `src/styles` -> `dist/content/styles`
4. copy `public` -> `dist/content/assets`
5. copy `.orbit/.sidebar/sidebar.css` -> styles output
6. generate sidebar config + flat page list
7. read all markdown files
8. parse frontmatter/body
9. render markdown with Shiki highlighting
10. compute route + relative link depth (`baseRel`)
11. build SEO object
12. render page HTML with `layout.js`
13. optionally minify HTML
14. write page to `<route>/index.html`
15. collect sitemap/feed/SEO audit data
16. generate root index + course indexes
17. write SEO outputs + report
18. enforce `strictSeo` if enabled

This is useful when debugging "why did this field or link end up like this?"

## 11) End-to-end change lifecycle (start to finish)

Use this for every meaningful change.

1. **Sync**
   - `git pull`
2. **Install/update deps if lockfile changed**
   - `npm install`
3. **Develop**
   - `npm run dev`
   - edit content/code
4. **Review output in browser**
   - `npm run preview`
5. **Quality checks**
   - `npm run lint`
   - `npm run seo:check`
6. **Final build**
   - `npm run build`
7. **Commit source changes**
   - include code/content/metadata updates
   - do not include `dist/` (ignored)
8. **Deploy**
   - publish `dist/` via your host/CD pipeline

## 12) Deployment playbooks

Orbit emits plain static files. Any static host works.

### Generic static hosting requirements

- publish directory: `dist`
- rewrite behavior: optional, since routes are folder-based (`/x/y/` -> `/x/y/index.html`)
- set correct MIME types for XML and CSS (most hosts do this automatically)

### Basic deployment checklist

- confirm `metadata.json.url` equals production domain
- run `npm run build`
- verify:
  - `dist/sitemap.xml`
  - `dist/robots.txt`
  - `dist/feed.xml`
  - key content routes
- upload/publish `dist`

## 13) Troubleshooting guide

### Pages not generated

Likely causes:

- files are outside `src/pages`
- extension is not `.md` or `.mdx`
- syntax error in markdown frontmatter block

Check:

- file path and extension
- YAML delimiters (`---`) are valid

### Sidebar order is incorrect

Check precedence chain:

1. frontmatter `order`
2. parent `_meta.json.items`
3. file/folder prefix

Also check for spelling mismatch in `_meta.json.items` keys (keys must match cleaned file names).

### SEO report has many warnings

Review `dist/seo-report.json` and prioritize:

- missing image
- missing canonical
- weak or too-short description

Fix with frontmatter first, then metadata defaults.

### Preview not reflecting changes

Ensure you rebuilt:

- `npm run build`

If watch mode is running, check terminal logs for build errors.

### Build fails with strict SEO

If `metadata.json.strictSeo` is `true`, any SEO warning fails the build.

Options:

- fix all warnings (recommended for production)
- temporarily set `strictSeo: false` during drafts

## 14) Safe contribution rules for this repo

- Keep `main.js`, `layout.js`, `utils.js`, and sidebar logic synchronized when changing routes or path math.
- When changing output structure, validate:
  - page links
  - pagination links
  - sidebar links
  - canonical URLs
  - sitemap links
- Avoid introducing host-specific assumptions in generated HTML unless required.
- Run lint + build + SEO check before opening a PR.

2. Run `npm install`.
3. Run `npm run build`.
4. Run `npm run preview` and open `http://127.0.0.1:5500`.
